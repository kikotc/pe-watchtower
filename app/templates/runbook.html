Alert: Service Down Critical
Triggered when GET /health fails or times out. Monitor checks every 15 seconds. Alert fires to Discord with a 5-minute cooldown between repeats.
  1. 1
    Check the monitor dashboard (port 5002)

    Open http://localhost:5002/dashboard. This runs independently of the main app — it stays up even when the app crashes. The status banner will show red if the app is unreachable.

  2. 2
    Check if the Flask process is running 
    terminal
    ps aux | grep run.py
docker ps  # check if postgres container is also up
  3. 3
    Read the crash reason from logs

    The monitor dashboard Live Log Feed reads directly from logs/app.log even when the app is down. Or read it directly:

    terminal
    tail -50 logs/app.log

    Look for ERROR or Exception lines just before the service went silent.

  4. 4
    Check the database is reachable 
    terminal
    docker ps | grep postgres
# if stopped:
docker start watchtower-db
    The /health endpoint runs SELECT 1 against Postgres on every call. A dead DB = a failing health check.
  5. 5
    Restart the Flask app

    See the Starting & Restarting section below. Once the app recovers, the monitor will automatically fire a ✅ Service Recovered Discord alert.

Alert: High Error Rate Warning
Triggered when ≥ 50% of requests in the last 2-minute window return 5xx, with at least 5 requests in that window.
  1. 1
    Check the Error Rate card and chart

    Go to the dashboard Overview tab. The Error Rate card turns red when elevated. Switch to the Error Rate History chart and use the 5m/15m/30m selector to see when it started.

  2. 2
    Identify failing paths in the log feed

    In the Data tab → Live Log Feed, look for rows with a red left border (ERROR level) and red 5xx status codes. Note which endpoints are failing and when they started.

  3. 3
    Check for a bad recent commit 
    terminal
    git log --oneline -10

    If a recent commit introduced the errors, roll it back:

    terminal
    git revert HEAD
python3 run.py
  4. 4
    Check database connectivity

    Most 5xx errors are DB failures. Confirm the container is running and the tables exist:

    terminal
    docker ps | grep postgres
# if tables are missing, re-run the seed:
python3 seed.py
Alert: Service Recovered Auto
Fired automatically by the monitor when /health returns 200 after a previous outage. Includes total downtime duration. No action required — this is informational.
After a recovery alert, check the Incident History panel on the dashboard to confirm the incident was recorded and review duration.

General Debugging Steps

  1. 1
    Check the Live Log Feed (Data tab)

    GET /logs returns the last 200 log entries in memory. Each request is logged with method, path, status code, and duration. Left border color = severity. Red border = error, amber = warning.

  2. 2
    Check system metrics (Overview tab rings)

    CPU and RAM rings glow red when above 75%. High CPU (>90%) or RAM saturation causes slow responses and eventually timeouts. Check GET /metrics for raw values.

  3. 3
    Check Top Endpoints (Data tab)

    Is one path receiving an abnormally disproportionate share of traffic? The gold bar = #1 endpoint. A runaway client, polling loop, or load test will show here immediately.

  4. 4
    Check historical charts for the onset time

    Use the 5m/15m/30m/1h time selector on the Overview charts. Error Rate and Response Time are seeded from the full log file, so you can see history even after a restart.

  5. 5
    Check Recent Events (Data tab)

    GET /events returns the last 100 data-mutation events. Unexpected mass-deletes or creates surface here with a red/green left border accent.

Starting & Restarting Everything

Three separate terminals. Start them in this order.

Terminal 1 PostgreSQL (Docker) 

first time only
docker run -d --name watchtower-db \
  -e POSTGRES_DB=hackathon_db \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 postgres:16
subsequent runs
docker start watchtower-db

Terminal 2 Flask App → localhost:5001 

seed DB first time only
python3 seed.py   # creates tables + inserts sample data
every time
python3 run.py

Terminal 3 Monitor + Dashboard UI → localhost:5002 

every time
python3 monitor.py
Confirm recovery: curl http://localhost:5001/health should return {"status":"ok","database":"ok"}.

SLO Breach Response

SLO target is 99.9% uptime (error budget = 0.1% of all requests). When budget consumed exceeds 100%, the SLO is breached.

  1. 1
    Check the SLO widget on the Overview tab

    The widget shows uptime %, error budget consumed, and total error requests. A red "✗ Breached" badge means the budget is exhausted.

  2. 2
    Check the Incident History panel

    The panel shows the last 20 "Alert fired" log entries. This tells you when and how many times the service degraded in the current log window.

  3. 3
    Reduce errors to recover budget

    Fix the root cause of 5xx responses (see High Error Rate guide above). The SLO widget recomputes live on the next poll — budget consumed will drop as new successful requests come in.

    SLO is computed from the full logs/app.log file. Deleting the log file resets all history including the SLO calculation.
